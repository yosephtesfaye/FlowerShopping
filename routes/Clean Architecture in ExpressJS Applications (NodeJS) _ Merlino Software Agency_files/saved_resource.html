                        Romeo Bellon Moderator Brett Bedarf 6 months ago edited

                        Ok, I understand what you mean, and you are right: we should indeed state in the article that we took the following trade-offs:
                        - Return ORM-specific models to use-cases when calling repo/data access methods to avoid having to define getters and setters for properties (example here);
                        - Use the entity layer as the abstraction layer between Mongoose (a "structuring" layer itself) and the use-cases, still in the NoSQL realm.
                        These trade-offs would still allow us to swap to other NoSQL databases or ORMs by changing the implementation of `findOne(),` `insert(),` etc. However, in the case of moving to SQL, then we would definitely need to change a few more things.
                        In summary, we preferred to deal with JSON objects in use-cases rather than having getters and setters everywhere, considering that Mongoose does the model validation.
                        I'd love to hear if that makes sense to you. Would you have done it differently?
